So why did I create this tool?


It started with me wanting to tweak a few values in my Frosthaven campaign. Maybe give myself a bit more gold, or adjust a character's stats. Sounds easy, right? Wrong.

The game's ruleset file is incredibly strict. This isn't the kind of file you can just open in Notepad, change a number, hit save, and call it a day. The game relies on the file having an exact byte structure. If that structure shifts, even by a single byte, the entire file becomes corrupted and the game simply won't load.

Now, you might say “Just don't add extra characters. If a value is one digit, keep it one digit and that should be fine, right?” Nope.

The problem is that characters and bytes are not the same thing. Modern text editors use UTF-8 encoding, which means that what looks like a single character can take up a different number of bytes depending on its value and context. Even changing a number from one digit to another, without adding or removing characters, can still change the file's byte size.

On top of that, some values can be stored as signed integers, others as unsigned. So a value that looks safe to change might require more bytes internally. This means you're never guaranteed a 1:1 byte replacement when editing the file in a normal text editor. Even if the visible text looks identical in length, the underlying byte data may not be.

That's why this can't reliably be done in Notepad or similar tools. To do it safely, you either need a hex editor, where you can see and control the exact byte values, or you need to edit the file at the byte level using a single-byte encoding like ISO-8859-1.

Once I understood this, I ran into the real challenge.

If increasing a value adds bytes to the file, those bytes have to be removed somewhere else. The total file size must remain exactly the same, and this has to be done without altering the files internal structure.

While examining the file in a hex editor, I noticed something interesting. Every line in the file ends with a Windows-style line break: 0D 0A. In most cases, only one of those bytes is actually necessary.

After testing, I confirmed that I could safely "steal" bytes by removing one of these line breaks to compensate for bytes added elsewhere.

Considering the file is around 250,000 lines long, with roughly 100,000 lines of editable content, there turned out to be plenty of bytes to work with.

That discovery is what made this tool possible.

I'll be honest, this project gave me more headaches than I care to admit. As a first-year student, this code became much more advanced than I was prepared for.

However, during this journey, I've learned a lot about file handling, byte conversions, and Java programming in general. I hope you find this program useful, and if you have any questions or suggestions, feel free to reach out on GitHub.
